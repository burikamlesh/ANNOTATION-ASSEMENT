# Architectural & Engineering Decisions Document

### 1. Data Flow & Normalization Strategy
* **Messy Payloads Handling:** Input dates dynamically map alternate epochs or ISO syntax variables cleanly through conditional parsing blocks inside `normalize.ts`. Stringified count structures are cast explicitly into strict integers (`Number()`). Missing fields default gracefully to fallback placeholders (e.g., `title: task.title || 'Untitled Task'`).
* **Redux Architecture:** Integrated standard `createEntityAdapter` within `taskSlice` which sets an optimal $O(1)$ access structure using entity lookup maps. Memoized filter selections perform atomic normalization checks (e.g., matching inconsistent casing blocks like `.toLowerCase() === status.toLowerCase()`) preventing costly UI recalculation pipelines.

### 2. Streamed Content Security Execution
* **Incremental Text Processing:** The custom stream processor `useTaskSummary` reads the SSE body with `response.body.getReader()`, buffers partial lines across chunk boundaries, and `JSON.parse`s each `data: "..."` frame (the mock server JSON-encodes each chunk so embedded `\n`s survive the SSE framing). Each parsed chunk is appended to `summary` state as it arrives, so `TaskSummary` re-renders incrementally rather than waiting for `event: done`.
* **XSS Sanitization Guard (corrected):** An earlier version of this file claimed sanitization happened via a custom `<ReactMarkdown components={{ html: ... }}>` override piped through `isomorphic-dompurify`. That was wrong and has been removed: `react-markdown`'s `components` map only intercepts *known HTML tag renderers* (`p`, `code`, `img`, etc.) produced from parsed markdown AST nodes — there is no `"html"` node type unless the `rehype-raw` plugin is added to convert embedded raw HTML into real hast elements. Since this project does **not** add `rehype-raw`, `react-markdown` never parses the mock server's `<script>`/`<img onerror>` payload as HTML at all — it's treated as literal text and HTML-escaped on output (`&lt;script&gt;...&lt;/script&gt;`). That's the actual sanitization boundary: **the absence of raw-HTML parsing**, not a DOMPurify pass. Verified in `src/components/summary/TaskSummary.security.test.tsx`, which renders the exact malicious payload from `mock-server/server.js` and asserts no `<script>`/`<img>` element exists in the resulting DOM. If a future requirement needs real embedded HTML (e.g. authoring bold/italic via raw tags), the correct fix is `rehype-raw` + `rehype-sanitize` (allowlist-based), not a hand-rolled DOMPurify call on a code path that never executes.

### 3. Real-Time Concurrency System
* **WebSocket Management Strategy:** The integration framework routes socket events through a specialized lifecycle client hook (`useTaskFeed`). To preserve atomic structural references, updates execute via selective patches (`updateTask`), preventing complete entity rewrites that could accidentally drop local UI data flags. Uncached rows incoming via event targets are queued or left safely until next fetch validation boundaries sync.

### 4. Code Bug Defect Registry (buggy/TaskTicker.tsx)
* **Bug 1 (Stale Closure):** `setInterval` was accessing an old fixed baseline lexical snapshot of the `tick` reference. *Fix:* Updated parameter loop to execute with a functional updater pattern expression (`setTick(prev => prev + 1)`).
* **Bug 2 (State Mutation via Push):** Arrays were being manipulated in-place (`prev.push`). This breaks React virtual identity tracking checks. *Fix:* Implemented explicit array array-spreading arrays (`[...prev, t]`).
* **Bug 3 (Null Fetch Operations):** Component sent API network requests on component initialization when `selectedId` was completely unassigned (`null`). *Fix:* Injected an active early exit validation block (`if (!selectedId) return`).
* **Bug 4 (Array Sort Side Effect):** Calling `.sort()` directly alters state vectors inline within the component view pipeline. *Fix:* Conducted sorting procedures on a isolated copy (`[...tasks].sort()`).
* **Bug 5 (Index Key Anti-pattern):** Component mapped rows using continuous indices (`key={i}`). *Fix:* Substituted layout mapping targets to reference structural item keys (`key={t.id}`).

### 5. Post-hoc cleanup (found during a self-audit before submission)
While reviewing the codebase against the spec, I found leftover code from an earlier approach that never got cleaned up, plus a couple of real bugs. Fixed all of these:

* **Dead, type-incompatible normalizer (`utils/normalizeTask.ts`)** — an earlier draft of the fetch path (`redux/tasks/taskThunk.ts`, since deleted) used a completely different, ad-hoc `normalizeTask` that mapped status to `Pending/Running/Completed/Failed` instead of the real `TaskStatus` union (`TODO/IN_PROGRESS/DONE/QA/BLOCKED/UNKNOWN`) used everywhere else (selectors, WebSocket events, `TaskToolbar`). It also typed its input as `raw: any`. It turned out to be dead code — the live app uses RTK Query (`taskApi.ts`) + `syncTasks.ts`, which calls the correct normalizer — but it still failed `tsc --noEmit` on its own and would have confused anyone reading the repo. Deleted `taskThunk.ts`, `utils/normalizeTask.ts`, the unused duplicate `types/normalize.ts`, and the matching dead `TaskFilters.tsx` component (superseded by `TaskToolbar.tsx`, which is what `page.tsx` actually renders).
* **Typo bug in the real normalizer** — `utils/normalize.ts` had `odo: "TODO"` instead of `todo: "TODO"` in the status map, so any task whose raw status was literally `"todo"` fell through to `UNKNOWN`. Fixed the key and extracted a standalone `normalizeStatus()` export so the WebSocket hook normalizes incoming `task.updated` status strings the same way the REST path does, instead of trusting the server's raw casing.
* **`annotation.created` overwrote instead of incremented** — `useTaskFeed.ts` dispatched `upsertTask({ annotationCount: 1, ... })` on every `annotation.created` event, which reset the count to `1` instead of incrementing it. Added an `incrementAnnotationCount` reducer to `taskSlice` that reads the existing entity and increments it in place (a no-op if the task isn't loaded yet, which satisfies the "event references a task you haven't loaded" requirement without extra bookkeeping).
* **Unjustified `as any` casts** — `useTaskFeed.ts` cast its dispatch payloads to `any` to dodge the `Update<Task>` shape mismatch, and `TaskRow.tsx` cast `task.assignee`/`task.annotationCount` to `any` even though both are already properly typed on `Task`. Replaced with `updateTask({ id, changes })` (the adapter's real `Update<Task, string>` type) and removed the redundant casts in `TaskRow.tsx`.
* **Unused `socket.io-client` dependency** — the mock server speaks plain `ws`, and `WebSocketManager` already used the native `WebSocket` API, so this dependency was never imported. Removed it and typed `WebSocketManager`'s message handler against the real `WsEvent` union instead of `any`.
* **Dead/misleading DOMPurify usage removed** — see the corrected §2 above. Removed the non-functional `html` component override from `TaskSummary.tsx` and dropped the now-unused `dompurify`/`isomorphic-dompurify`/`@types/dompurify` dependencies.
* **More unjustified `any` casts (`TaskDrawer.tsx`)** — it read `(task as any).meta?.priority` and `(task as any).note`, checking for a `meta` property that doesn't exist on `Task` at all (a leftover from the raw/pre-normalized shape). `Task` already has `priority`/`note` as first-class optional fields post-normalization; replaced with plain `task.priority` / `task.note`.
* **Generic README replaced** — it was still the unmodified `create-next-app` boilerplate. Rewrote it with actual run instructions (mock server + app, in order), test command, required env vars, and a pointer to this file.
* **`.next/` was tracked in git** despite being in `.gitignore` (188 files, added before the ignore rule existed), causing noisy diffs on every dev run. Ran `git rm -r --cached .next` and added `*.tsbuildinfo` to `.gitignore` as well.

### 6. Tests
Added Jest + React Testing Library via `next/jest` (SWC transform). One non-obvious wrinkle: `next/jest`'s default `transformIgnorePatterns` excludes all of `node_modules`, including `react-markdown` and its ESM-only dependency tree (`unified`/`remark`/`hast`/`mdast`/`micromark`/...), and it *concatenates* rather than overrides when you pass your own pattern. `jest.config.js` works around this by resolving `next/jest`'s config first and then explicitly replacing `transformIgnorePatterns` on the resolved object. Four suites, covering the spec's minimum plus one extra for the security claim in §2:
* `utils/normalize.test.ts` — the normalizer: mixed epoch/ISO timestamps resolve to the same value, string counts coerce to numbers, unknown types map to `"unknown"` instead of being dropped, null assignee is preserved.
* `redux/tasks/taskSelectors.test.ts` — `selectFilteredTasks`: status filtering and case-insensitive search narrow the result set correctly.
* `components/task/TaskFiltering.test.tsx` — RTL interaction test: renders `TaskToolbar` + `TaskTable` against a real store, selects a status filter, asserts the visible rows change.
* `components/summary/TaskSummary.security.test.tsx` — renders the mock server's exact `<img onerror>`/`<script>` payload and asserts no such element exists in the DOM, proving the claim in §2 rather than just asserting it in prose.

### 7. AI usage disclosure
This codebase was built with heavy AI assistance (Claude), including the initial scaffolding of the Redux slice/adapter/selectors, the normalizer, the WebSocket hook, and the summary streaming/sanitization path. Verification approach:
* Every non-obvious piece was read line-by-line and cross-checked against what the mock server actually sends (`mock-server/server.js`), not just against the prose spec.
* Before this cleanup pass, I ran a structural audit comparing every file against the spec's explicit requirements, which is what surfaced the dead/duplicate normalizer, the `annotationCount` overwrite bug, and the unjustified `any` casts documented above — none of these were caught by `tsc` or the app running normally, since the broken code path was unreachable (dead) or silently wrong (overwrite) rather than crashing.
* Added `tsc --noEmit` and the full Jest suite as a hard gate after every change in this pass; both are green as of this commit.
* I did not blindly keep AI-authored code that I couldn't explain: the deleted `taskThunk.ts`/`normalizeTask.ts` pair is exactly the kind of plausible-looking but wrong output I'm on guard for, and it's why I made a deliberate pass to find and remove it rather than assuming "it compiles somewhere" meant "it's correct."
